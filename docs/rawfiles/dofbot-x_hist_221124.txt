  199  ## Changing the Base Image
  200  By default, the base container image used at the start of the build chain will be [`l4t-base`](https://catalog.ngc.nvidia.com/orgs/nvidia/containers/l4t-base) on JetPack 4, [`l4t-jetpack`](https://catalog.ngc.nvidia.com/orgs/nvidia/containers/l4t-jetpack) on JetPack 5, and [`ubuntu:22.04`](https://hub.docker.com/_/ubuntu/tags?page=&page_size=&ordering=&name=22.04) on JetPack 6.  However, if you want to add packages to a container that you already have, you can specify your own base image:
  201  ```bash
  202  jetson-containers build --base=my_container:latest --name=my_container:pytorch pytorch  # add pytorch to your container
  203  ```
  204  > [!NOTE]  
  205  > On JetPack 4/5, it's assumed that base images already have the JetPack components available/installed (CUDA Toolkit, cuDNN, TensorRT).  
  206  > 
  207  > On JetPack 6 (L4T r36.x), the CUDA components will automatically be installed on top of your base image if required. 
  208  > > You can specify the CUDA version by using `CUDA_VERSION` environment variable, otherwise it will pick the CUDA version that was made available to the L4T version.
  209  ## Changing Versions
  210  Many packages are versioned, and define subpackages like `pytorch:2.0`, `pytorch:2.1`, ect in the package's [configuration](/docs/packages.md#python).  For some core packages, you can control the default version of these with environment variables like `PYTORCH_VERSION`, `PYTHON_VERSION`, and `CUDA_VERSION`.  Other packages referring to these will then use your desired versions instead of the previous ones:  
  211  ```bash
  212  CUDA_VERSION=12.4 jetson-containers build --name=cu124/ pytorch  # build PyTorch for CUDA 12.4
  213  ```
  214  If you omit a tag from your name, then a default one will be assigned (based on the tag prefix/postfix of the package - typically `$L4T_VERSION`)
  215  You can also build your container under a namespace ending in `/` (for example `--name=my_builds/` would result in an image like `my_builds/pytorch:r36.2.0`).  Using namespaces can help manage/organize your images if you are building a lot of them.
  216  ## Multiple Containers
  217  The `--multiple` flag builds separate containers for each of the packages specified:
  218  ```bash
  219  $ jetson-containers build --multiple pytorch tensorflow2   # built a pytorch container and a tensorflow2 container
  220  $ jetson-containers build --multiple ros*                  # build all ROS containers
  221  ```
  222  The dependencies are also able to specify with [`requires`](/docs/packages.md) which versions of L4T, CUDA, and Python they need, so changing the CUDA version has cascading effects downstream and will also change the default version of cuDNN, TensorRT, and PyTorch (similar to how changing the PyTorch version also changes the default version of torchvision and torchaudio).  The reverse also occurs in the other direction, for example changing the TensorRT version will change the default version of CUDA (unless you explicitly specify it otherwise).  
  223  | Package       | Environment Variable | Versions                                                                             |
  224  |---------------|----------------------|-----------------------------------------------------------------------------------------|
  225  | [`cuda`](/packages/cuda/cuda) | `CUDA_VERSION` | [`cuda/config.py`](/packages/cuda/cuda/config.py) |
  226  | [`cudnn`](/packages/cuda/cudnn) | `CUDNN_VERSION` | [`cudnn/config.py`](/packages/cuda/cudnn/config.py) |
  227  | [`tensorrt`](/packages/tensorrt) | `TENSORRT_VERSION` | [`tensorrt/config.py`](/packages/tensorrt/config.py) |
  228  | [`python`](/packages/python) | `PYTHON_VERSION` | [`python/config.py`](/packages/python/config.py) |
  229  | [`pytorch`](/packages/python) | `PYTORCH_VERSION` | [`pytorch/config.py`](/packages/pytorch/config.py) |
  230  Using these together, you can rebuild the container stack for the specific version combination that you want:
  231  ```bash
  232  CUDA_VERSION=12.4 PYTHON_VERSION=3.11 PYTORCH_VERSION=2.3 \
  233    jetson-containers build l4t-text-generation
  234  ```
  235  The available versions are defined in the package's configuration scripts (some you can simply add new releases to by referring to the new git tag/branch, others need pointed to specific release downloads).  Not all combinations are compatible, as the versioned packages frequently define the minimum version of others in the build tree they rely on (for example, TensorRT 10 requires CUDA 12.4).
  236  For packages that provide different versions but don't have their own environment variable defined, you can specify the desired version of them that your container depends on in the Dockerfile header under [`depends`](/docs/packages.md), and it will override the default and build using them instead (e.g. by using `depends: onnxruntime:1.17` instead of `depends: onnxruntime`).  Like above, these versions will also cascade across the build.
  237  > [!NOTE]  
  238  > On JetPack 6 (L4T r36.x), if you don't have `CUDA_VERSION` specified, it will pick the CUDA version that was packaged with that version of L4T in the JetPack.
  239  > - L4T r36.2 (JetPack 6.0 DP) --> CUDA `12.2`
  240  > - L4T r36.3 (JetPack 6.0 GA) --> CUDA `12.4`
  241  > - L4T r36.4 (JetPack 6.1) --> CUDA `12.6`
